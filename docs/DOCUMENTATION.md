## 1. Architecture overview

The project follows a layered, "four-tier" style architecture meant to separate concerns as the codebase grows: **presentation → application → domain → infrastructure**. The intent is that each layer has one job, so a change in one (e.g., swapping Postgres for another database, or REST for GraphQL) doesn't ripple through the others.

```
┌───────────────────────────────────────────────────────────────┐
│  Presentation  (backend/app)                                  │
│  FastAPI routers, Pydantic schemas, WebSocket/Socket.IO,      │
│  middleware, the app factory                                  │
└───────────────────────────┬───────────────────────────────────┘
                             │  calls
┌───────────────────────────▼───────────────────────────────────┐
│  Application  (backend/application)                           │
│  Use cases (RegisterUserUseCase, LoginUseCase, ...),          │
│  ports (interfaces for outbound integrations)                 │
└───────────────────────────┬───────────────────────────────────┘
                             │  calls
┌───────────────────────────▼───────────────────────────────────┐
│  Domain  (backend/domain)                                     │
│  Services (UserService, ProductService) — business rules,     │
│  Repositories — persistence access,                           │
│  Models — SQLAlchemy entities,                                │
│  Events — DomainEvent for meaningful state changes            │
└───────────────────────────┬───────────────────────────────────┘
                             │  uses
┌───────────────────────────▼────────────────────────────────────┐
│  Infrastructure  (backend/database, backend/infrastructure,    │
│  backend/utils, backend/integrations, backend/core/security,   │
│  backend/resilience, backend/observability, backend/common)    │
│  DB engine/session, Redis client, email transport, runtime     │
│  wiring, shared cross-cutting helpers (logging, rate limiting, │
│  auditing, tracing, exceptions)                                │
└────────────────────────────────────────────────────────────────┘
```

A request typically moves through the stack as follows:

1. **Middleware** (`backend/main.py`) assigns/propagates a request ID and trace ID, enforces rate limiting and max request size, and wraps the call in a trace span.
2. **The API router** (`backend/app/api/v1/...`) receives the request and its dependencies resolve (current user via JWT, DB session, service instances).
3. **The application layer** (a use case in `backend/application/<feature>/use_cases.py`) coordinates the workflow: validates business preconditions, calls the domain layer, and translates domain errors into a result.
4. **The domain layer** (`backend/domain/<feature>/service.py` + `repository.py`) applies business rules and persists/loads data. It emits a `DomainEvent` when something significant happens (currently: user registration).
5. **The infrastructure layer** underneath handles the actual DB/Redis/SMTP calls, plus cross-cutting concerns like audit logging and metrics.
6. **The response** is translated back into a Pydantic schema (`backend/common/schema.py`) and returned; the middleware attaches security headers and records metrics on the way out.

## 2. Request lifecycle, in detail

This section traces a single `POST /api/v1/products/` call all the way through, as a concrete example of how the layers connect.

1. **Uvicorn → FastAPI middleware stack.** `backend/app/bootstrap/middleware_registry.py` has already installed `CORSMiddleware`. `backend/main.py`'s `add_request_context` middleware then runs for every request:
   - Reads/generates `x-request-id` and `x-trace-id`.
   - Calls `bind_request_context(...)` (`backend/observability/logging.py`) so every log line for this request carries the request ID.
   - If `settings.enable_rate_limiting` is on, checks `shared_rate_limiter.allow_request(...)` (`backend/resilience/rate_limit.py` — see §11 for a note on a stale duplicate of this module) and short-circuits with `429` if the caller has exceeded `rate_limit_requests_per_minute`.
   - A separate ASGI middleware, `backend/web/request_size_middleware.py`, checks `Content-Length` against `settings.max_request_size_bytes` and short-circuits with `413` if too large.
   - Wraps the actual route call in `trace_span("http.request", ...)` (`backend/observability/tracing.py` — this is the sole tracing module; `backend/common/opentelemetry.py` does not exist in this codebase despite being referenced elsewhere in this project's history).
2. **Routing.** `backend/app/bootstrap/router_registry.py` mounted `backend/app/api/v1/router.py` at `settings.api_v1_str` (default `/api/v1`), which in turn includes `backend/app/api/v1/products/router.py`'s `create_product` handler for this path.
3. **Dependency resolution.** FastAPI resolves `db: AsyncSession = Depends(get_db)` (`backend/database/session.py`), opening an `AsyncSession` that will be committed/rolled back automatically at the end of the request.
4. **Application layer.** The handler builds `ProductService(ProductRepository(db))` and calls `CreateProductUseCase(service).execute(payload=payload)` (`backend/application/products/use_cases.py`), which validates uniqueness rules and calls into the domain service.
5. **Domain layer.** `ProductService.create(...)` (`backend/domain/products/service.py`) constructs the `Product` SQLAlchemy model, adds it to the session, flushes, and refreshes it to get the generated `id`.
6. **Response translation.** The router converts the returned `Product` into `ProductOut` (`backend/common/schema.py`) via `response_model=ProductOut`.
7. **Audit + metrics on the way out.** The router calls `audit_logger.log(...)` (`backend/observability/audit.py`); back in the middleware, `record_request_metrics(...)` (`backend/observability/metrics.py`) records the method/path/status, and security headers (`x-content-type-options`, `x-frame-options`, `referrer-policy`, and — if `require_https` — `strict-transport-security`) are attached to the response.
8. **Session teardown.** `get_db()`'s `finally` block closes the session; if an exception propagated, it's rolled back first.

The same shape applies to every feature — only steps 4–5 change per domain.

## 3. Layer-by-layer guide

### 3.1 Presentation layer — `backend/app`

Responsible for the HTTP/WebSocket surface of the app.

| Concern                                                                     | Where                                                                                                                          |
| --------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| App factory (`create_app()`)                                                | `backend/app/factory.py`                                                                                                       |
| Middleware registration (CORS)                                              | `backend/app/bootstrap/middleware_registry.py`                                                                                 |
| Router registration                                                         | `backend/app/bootstrap/router_registry.py`                                                                                     |
| Upload directory creation (not a static file mount — see note below)        | `backend/app/bootstrap/static_registry.py`                                                                                     |
| Infrastructure wiring hook-up for the factory                               | `backend/app/infrastructure/__init__.py` (re-exports `build_infrastructure_registry` from `backend/infrastructure/runtime.py`) |
| Top-level router aggregation                                                | `backend/app/api/v1/router.py`                                                                                                 |
| Feature routers                                                             | `backend/app/api/v1/{auth,users,products,uploads,admin,rbac,billing}/router.py`                                                |
| Socket.IO server & handlers                                                 | `backend/app/socketio_app.py`                                                                                                  |
| App entrypoint: middleware, `/health`, `/metrics`, `/runtime`, `/ws/health` | `backend/main.py`                                                                                                              |

### 3.2 Application layer — `backend/application`

Where business workflows become explicit, orchestrated units instead of logic embedded in route handlers.

| Concern                                                                     | Where                                                                                        |
| --------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------- |
| User registration & update workflows                                        | `backend/application/users/use_cases.py` (`RegisterUserUseCase`, `UpdateUserUseCase`)        |
| Auth workflows (login, refresh, logout, password reset, email verification) | `backend/application/auth/use_cases.py`                                                      |
| Product create/update workflows                                             | `backend/application/products/use_cases.py` (`CreateProductUseCase`, `UpdateProductUseCase`) |
| Outbound integration interfaces ("ports")                                   | `backend/application/ports.py` (`NotificationPort`)                                          |

Each use case takes the domain service(s) it needs as constructor arguments, does precondition checks (e.g., "does this email already exist?"), delegates the actual state change to the domain layer, and returns a result the router can translate into a response — keeping routers limited to request parsing, calling the use case, and formatting the response/errors.

### 3.3 Domain layer — `backend/domain`

| Concern                                                      | Where                                                                 |
| ------------------------------------------------------------ | --------------------------------------------------------------------- |

| Product entity                                               | `backend/domain/products/model.py`                                    |
| Product business rules                                       | `backend/domain/products/service.py`                                  |
| Product persistence (search/sort/filter)                     | `backend/domain/products/repository.py`                               |
| Domain events                                                | `backend/domain/events/__init__.py` (`DomainEvent`)                   |
| Shared generic repository/service base classes               | `backend/common/base_repository.py`, `backend/common/base_service.py` |

A previous revision of this table incorrectly marked `GET`/`DELETE /api/v1/users/{user_id}` as unauthenticated; that was never accurate for the current router implementation (verified directly against `backend/app/api/v1/users/router.py`) and has been corrected above.

The `require_role("admin")`/`require_role("admin", "staff")` dependency (`backend/core/security/rbac.py`) checks `current_user.role` (or `current_user.is_superuser`) against the given role names, and is wired into every user-listing and product-write route. A separate, finer-grained `require_permission(*permission_keys)` dependency does exist in the same module, checking `current_user.permissions` (or `is_superuser`) against a permission key rather than a role name — it's what gates every `billing.manage` route in §4/§14 and the RBAC-management routes themselves (now listed as full table rows above, rather than only in prose). There genuinely is no `AuthorizationPolicy`/`require_policy`-based route in the current code — `require_policy` exists in `rbac.py` but nothing calls it.

## 5. Configuration & environment variables reference

All settings are defined in `backend/core/config.py` (`Settings`, loaded via `pydantic-settings`), sourced from (in order of precedence) `ENV_FILE` if set, then `.env`, `.env.<environment>`, `.env.<environment>.local`, `.env.local`, then real process environment variables, then field defaults.

| Variable                                                                    | Default                                                       | Purpose                                                                                                                                                                                                                                      |
| --------------------------------------------------------------------------- | ------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `project_name`                                                              | `Tier 4 Architecture`                                         | OpenAPI title.                                                                                                                                                                                                                               |
| `api_v1_str`                                                                | `/api/v1`                                                     | Prefix for all v1 routes.                                                                                                                                                                                                                    |
| `environment` / `APP_ENV`                                                   | `dev`                                                         | One of `dev`, `staging`, `prod`.                                                                                                                                                                                                             |
| `DATABASE_URL`                                                              | `postgresql+asyncpg://postgres:postgres@localhost:5432/tier4` | Async SQLAlchemy connection string.                                                                                                                                                                                                          |
| `DATABASE_URL_FILE`                                                         | —                                                             | Path to a file containing the DB URL (for secret-mount deployments); overrides `DATABASE_URL` if the file exists.                                                                                                                            |
| `REDIS_URL`                                                                 | `redis://localhost:6379/0`                                    | Redis connection string.                                                                                                                                                                                                                     |

## 6. How to implement a new feature (step-by-step)

Every new feature should follow the same layered pattern so the app stays predictable and testable.

1. **Domain model.** Add the entity under `backend/domain/<feature>/model.py`. Keep it focused on the business concept — no HTTP/infrastructure concerns.
2. **Persistence.** Add `backend/domain/<feature>/repository.py`, inheriting `BaseRepository` for the common `get_by_id`/`list`/`count` operations, and add feature-specific queries (filters, search, sort) alongside them.
3. **Domain service.** Add `backend/domain/<feature>/service.py` for the actual business rules — validation, state transitions, anything that must always hold true regardless of which caller triggers it.
4. **Application use case(s).** Add `backend/application/<feature>/use_cases.py` for the orchestration: precondition checks that need to look across services (e.g., "does this email already exist"), calling the domain service, and returning a result. This is what keeps route handlers thin.
5. **Ports (only if there's an outbound integration).** If the workflow needs to call an external system (email, SMS, a third-party API), define a small `Protocol` in `backend/application/ports.py` describing only the methods the use case needs, and implement it under `backend/integrations/<integration>_adapter.py`. Keep the adapter the only place that imports the third-party SDK.
6. **Should this run asynchronously?** If the work is slow, best-effort, or shouldn't block the response (sending a notification, exporting data), hand it to `background_job_manager` (`backend/common/background_jobs.py`) rather than awaiting it inline in the use case — see the existing `enqueue_billing_notification` call sites for the pattern. **Be aware of the limitation this implies:** today `background_job_manager` runs jobs as in-process `asyncio` tasks with no persistence — a process restart mid-job silently drops the work (documented in §11.1). If the feature you're building genuinely needs at-least-once delivery (e.g., it must eventually succeed even across a restart), that's a real gap right now, not something to work around ad hoc inside the feature itself; flag it rather than inventing a one-off durability mechanism. A durable, Redis-backed replacement for `background_job_manager` is planned — see `docs/IMPLEMENTATION_PLAN.md` — but as of this revision it does not exist in the codebase yet, so don't write code or docs that assume a `job_queue.enqueue(...)`-style API is already there.
7. **Domain events (optional).** If the workflow represents a meaningful business change other parts of the system might care about later (analytics, notifications, audit), emit a `DomainEvent.create({...})` via `platform_runtime.event_bus.publish(event)` (`backend/domain/events/__init__.py` for the type, `backend/infrastructure/runtime.py`'s `PlatformRuntime.event_bus` for the actual instance the app runs on) from the use case — see `backend/application/users/use_cases.py` for a real call site.
8. **Reacting to an event someone else emits (optional, the other half of step 7).** A feature doesn't have to be the one _emitting_ an event to care about it — it can also _subscribe_. Call `platform_runtime.event_bus.subscribe("some.event_type", your_async_handler)` during startup wiring, the same way `PlatformRuntime._register_default_handlers()` already subscribes a logging handler to `"user.registered"` (`backend/infrastructure/runtime.py`). That one existing subscriber only logs; nothing in the current codebase reacts to an event by doing real work (sending a notification to a third party, writing a second record, etc.) — if your feature is the first to do that, it's genuinely new ground for this codebase, not a well-worn path, so keep the handler small and make sure a failure inside it can't take down the request that published the event (`EventBus.publish` currently awaits handlers inline — a slow or raising handler affects the publisher directly, which is worth knowing before you put anything nontrivial in one).
9. **Schemas.** Define request/response models in `backend/common/schema.py` (or a feature-local file if it's getting large). Be deliberate about which fields are client-writable — see §11.2 for why this matters.
10. **Router.** Add `backend/app/api/v1/<feature>/router.py`. Keep it to: parse the request, resolve dependencies (DB session, current user, service), call the use case, translate the result/errors into a response. Decide explicitly whether each route needs `Depends(get_current_active_user)` and/or `Depends(require_role(...))` — don't rely on "the other routes in this file have it so this one probably does too."
11. **Register the router** in `backend/app/api/v1/router.py`.
12. **Tests.** Add both a unit test for the use case/service logic and an integration test (via `TestClient`) that exercises the real route, including a negative case for missing auth if the route is protected.

## 7. Runtime, observability, and operational endpoints

- `GET /health` — returns `{status, environment, version}`. Suitable for a liveness probe.
- `GET /health/ready` — pings the DB (`SELECT 1`) and Redis (`PING`); returns `ready` if both succeed, `degraded` otherwise. Suitable for a readiness probe.
- `GET /metrics` — returns an in-process snapshot of request counts, status codes, methods, and paths, collected by `backend/observability/metrics.py`'s `MetricsCollector`. This is process-local, in-memory, and resets on restart — it is not a Prometheus-scrape-compatible endpoint (no `text/plain` exposition format) and does not aggregate across multiple workers/pods. **It requires an authenticated admin caller** (`Depends(require_role("admin"))` in `backend/main.py`).
- `GET /runtime` — returns an operational snapshot (service name, environment, uptime, the same metrics snapshot) via `PlatformRuntime` (`backend/infrastructure/runtime.py` — not a separate `backend/platform/` package). **Also requires an authenticated admin caller.**
- Every response carries `x-request-id`, `x-trace-id`, `x-content-type-options: nosniff`, `x-frame-options: DENY`, and `referrer-policy: strict-origin-when-cross-origin`; if `require_https` is enabled, also `strict-transport-security`.
- Tracing: `backend/observability/tracing.py`'s `trace_span` context manager wraps the whole HTTP request in `main.py`, and several individual routes (`auth.login`, `product.create`, `upload.create`) wrap their own operation in a span too. Today this only produces structured log lines (see §11 for the gap between this and a real OpenTelemetry exporter pipeline).

## 8. Persistence and data layer

- Async SQLAlchemy engine/session setup lives in `backend/database/session.py`; `backend/infrastructure/persistence/session.py` re-exports the same objects for infrastructure-layer imports.
- `Base` (SQLAlchemy `DeclarativeBase`) is shared across all models, defined in `backend/database/base.py`.
- Repositories inherit `BaseRepository` (`backend/common/base_repository.py`) for `get_by_id`, `list`, `create`, `update`, `delete`, `count` — though note that `UserService`/`ProductService` currently implement their own create/update logic directly rather than calling the base repository's, for reasons tied to the extra fields they need to set (hashed passwords, timestamps).
- Migrations: `alembic/versions/20260704_initial_schema.py` is the initial migration, creating the `users` and `products` tables — as first written it was missing `users.is_verified`, `users.failed_login_attempts`, and `users.locked_until`, and typed `created_at`/`updated_at` as `VARCHAR(50)` instead of a timestamp. **This is already corrected by the follow-up migration `alembic/versions/20260707_add_missing_user_columns.py`**, which adds the three missing columns and converts both timestamp columns to real `DateTime` columns. `alembic upgrade head` was run against a throwaway database as part of this audit and completes cleanly, producing a schema that matches the `User`/`Product` models. Whenever you add or change a model going forward, generate/hand-write a matching migration and keep `tests/test_migrations_match_models.py`'s `test_alembic_upgrade_head_runs_cleanly_against_a_throwaway_database` test passing — the app's `lifespan` (`backend/app/lifespan.py`) also calls `Base.metadata.create_all` at startup as a convenience for local development, but that is not a substitute for migrations in any shared environment.
- `backend/scripts/seed_data.py` creates a default admin account (from the `default_admin_*` settings) and two sample products, if they don't already exist. Intended for local development only.

## 10. Deployment

- **Dockerfile**: multi-stage-free, single `python:3.12-slim` build; installs `requirements.txt`, copies the app, creates a dedicated non-root `app` user (`addgroup`/`adduser --system`), runs as that user, and defines a `HEALTHCHECK` that curls `/health`. Entrypoint is `uvicorn backend.main:app --host 0.0.0.0 --port 8000`.
- **docker-compose.yml**: brings up the API alongside `postgres:16-alpine` and `redis:7-alpine` for local/integration use. It requires real `POSTGRES_PASSWORD`/`REDIS_PASSWORD` values via `${POSTGRES_PASSWORD:?set a real password in .env}`/`${REDIS_PASSWORD:?set a real password in .env}` — there is no hard-coded `postgres`/`postgres` default, and Redis is started with `--requirepass`, so the compose file will refuse to start without both passwords supplied.
- **CI: exists at `.github/workflows/ci.yml`**, running on push/PR to `main`/`master`: lint (`ruff check backend tests`) → real `alembic upgrade head` against a Postgres service container → `pytest -q` → `scripts/security_scan.sh`. `aiosqlite` is declared in `pyproject.toml`'s `dev` extra, but that alone doesn't make a fresh install's test step pass out of the box — see §11.1 for `email-validator`/`python-multipart`/`prometheus_client`/`pytest-asyncio` gaps found in this pass. The **lint step still fails** as of this revision — `ruff check backend tests` reports **2** errors (down from a previously-reported 12, itself down from 24): an unused `BillingService` import in `backend/application/users/use_cases.py` and an unused `pytest` import in `tests/test_billing_hardening.py`, both resolved automatically by `ruff check backend tests --fix`. This would still block the later CI steps from running, but it's now a two-line, test-and-application-code cleanup rather than the larger issue previously documented.
- **Environment files**: `deployment/env/{dev,staging,prod}.env` hold environment-specific defaults; production secrets (DB URL, Redis URL, JWT secret) should be supplied via the `*_FILE` settings (pointing at a mounted secret file) rather than committed to any `.env`.
- **scripts/security_scan.sh**: runs `safety` (dependency CVE scan) and `bandit` (static security lint) and **does gate the build** — it sets `status=1` on either tool finding something and `exit 1`s unless `SECURITY_SCAN_SOFT_FAIL=1` is set, which CI never sets. It is wired into CI as the last step of `ci.yml`.

## 11. Known limitations & security notes (read before deploying)

This project is a boilerplate, and boilerplates get copy-pasted wholesale into real products more often than they get read line-by-line first. **This section reflects a direct re-verification performed against this source tree as of July 2026** — every claim below was checked by reading the relevant code and, where applicable, running it (migrations, the full `pytest` suite, `ruff`), not by trusting a previous revision of this document.

### 11.1 What's still genuinely open

- **CI lint step currently fails, but the error count previously documented here is stale:** `ruff check backend tests` — the first step of `.github/workflows/ci.yml` — now reports **2** errors (down from a previously-reported 12, which was itself down from 24), both auto-fixable with `ruff check backend tests --fix`: an unused `BillingService` import in `backend/application/users/use_cases.py`, and an unused `pytest` import in `tests/test_billing_hardening.py`. The previously-reported "`client` fixture redefined in `test_authorization_and_roles.py`" issue no longer exists in the current source — that file was cleaned up between the last audit pass and this one. This still blocks the rest of the CI pipeline from running, but it's now a two-line fix.
- **Two of `pyproject.toml`'s declared runtime dependencies are actually imported but missing from the `[project.dependencies]` list, so `pip install -e .` (without `.[dev]`, and even with it) does not reliably pull them in:** `email-validator` (needed for Pydantic's `EmailStr` validation in `backend/common/schema.py`) and `python-multipart` (needed by FastAPI's `Form(...)`/multipart upload handling in the auth and uploads routers) are both listed in `requirements.txt` but absent from `pyproject.toml`'s `dependencies` array. A fresh `pip install -e .` reproducibly fails at import time with `ImportError: email-validator is not installed...` and then, once that's patched, `RuntimeError: Form data requires "python-multipart" to be installed` — verified directly in this pass. Anyone installing via `pyproject.toml` alone (the more standard path for an installable package) hits both.
- **`prometheus_client` is imported but declared in neither `requirements.txt` nor `pyproject.toml`:** `backend/observability/metrics.py` imports `CollectorRegistry`, `Counter`, and `generate_latest` from `prometheus_client` to back `get_prometheus_metrics()` (the `/metrics` endpoint's `text/plain` response). Neither manifest lists it. A fresh install from either file fails at import time until `prometheus_client` is installed manually — verified in this pass.
- **`pytest-asyncio` is used but not declared as a dev dependency:** `tests/test_authorization_and_roles.py` and `tests/test_payment_adapters.py` mark several tests with `@pytest.mark.asyncio`, but `pytest-asyncio` isn't in `pyproject.toml`'s `dev` extra alongside `pytest`/`ruff`/`mypy`/`aiosqlite`. Without it, those async tests are silently skipped with a `PytestUnknownMarkWarning` rather than actually run — a fresh `pip install -e .[dev] && pytest -q` looks green while exercising less than the suite intends.
- **Neither manifest pins an upper bound on any dependency, and that's now causing real breakage on a fresh install:** as of this pass (July 2026), an unpinned `pip install` resolves `starlette` to `1.3.1` (a new major version) and `aiosqlite` to `0.22.1`. The former requires `httpx` to be present for `starlette.testclient` to import at all (previously an implicit transitive dependency that's since become an explicit runtime check). The latter causes `sqlite3.OperationalError: cannot commit transaction - SQL statements in progress` under the test suite's concurrent-session usage, which reproducibly fails 4–5 tests in `tests/test_billing_hardening.py` and `tests/test_rbac_and_subscription_regression.py` today even though the same tests pass under older `aiosqlite` releases. None of this is visible from reading the code — it only shows up by actually installing and running the suite fresh, which is what this pass did.
- **Rate limiting has a single-process ceiling outside of Redis:** `RedisRateLimiter` (`backend/resilience/rate_limit.py`) is used whenever `environment != "dev"` and Redis is reachable, and falls back to the in-process `RateLimiter` if Redis is unreachable (failing _open_ to degraded limiting, not a full outage) — the only remaining multi-instance gap is `dev` mode or a sustained Redis outage.
- **Tracing and metrics are lightweight by design:** `backend/observability/tracing.py` produces structured log lines gated by `OTEL_MODE`/`ENABLE_TRACING`, not a real `opentelemetry` SDK span exporter, even though the `opentelemetry-*` packages are dependencies and `OTEL_EXPORTER_OTLP_ENDPOINT` exists as a setting. `/metrics` and `/runtime` are process-local snapshots (though both are admin-authenticated today) that reset on restart and don't aggregate across workers/replicas.
- **Remote upload backends can't be downloaded through the API yet:** `S3UploadStorage`/`AzureUploadStorage` (`backend/infrastructure/upload_storage.py`) can already _store_ files, but `GET /api/v1/uploads/{stored_name}` explicitly returns `501` whenever `UPLOAD_BACKEND` isn't `local` — the route comment says to issue a short-lived signed URL from the storage provider instead, but nothing currently does that automatically.
- **The audit log and rate limiter are still per-process/per-instance stores** (`logs/audit.jsonl` for the former, Redis-or-in-memory for the latter) rather than a queryable, centrally-aggregated store — fine for a single instance or small deployment, a gap to close before serious horizontal scaling.

### 11.2 What's already fixed (verify before re-fixing)

A previous revision of this document — and, in a few cases, this section itself — described the items below as open. Each was re-checked directly against the current source and/or exercised via the test suite and found **already resolved as of this revision**:

- **`aiosqlite` is declared as a dependency:** `pyproject.toml`'s `dev` extra now includes `aiosqlite>=0.19.0` alongside `pytest`/`ruff`/`mypy`. This is still true and still necessary — but on its own it's no longer sufficient for a clean `pip install -e .[dev] && pytest -q`; see the new dependency-manifest and unpinned-version findings in §11.1 above for what else is missing or has since drifted.
- **The retry/circuit-breaker primitive is wired in:** `backend/infrastructure/email/transport.py`'s `SMTPEmailTransport` now constructs a `CircuitBreaker(failure_threshold=3, reset_timeout=60.0)` from `backend/resilience/retry.py` and catches `CircuitBreakerOpenError` around its send path — it's no longer unit-tested-but-unused.
- **`BaseRepository` no longer accepts a raw `dict`:** `create()` (`backend/common/base_repository.py`) requires either a model instance or an object with `model_dump()` (i.e., a Pydantic schema), and `update()` requires a schema with `model_dump()`, raising `TypeError` otherwise — the mass-assignment path via an untyped dict is closed.
- **`staging` is covered by the insecure-default guard:** `Settings._reject_insecure_prod_defaults()` in `backend/core/config.py` now checks `self.environment in ("prod", "staging")`, refusing to boot with the default `SECRET_KEY` or `DEFAULT_ADMIN_PASSWORD` in either environment, not just `prod`.
- Login lockout returns the identical `"Incorrect email or password"` 401 for an unknown email, a wrong password, and a locked account — no enumeration signal.
- The Socket.IO `connect` handler checks the same Redis-backed revocation store that `/auth/logout` writes to, so a revoked access token can't open a realtime session.
- File uploads use server-generated UUID filenames with path-traversal protection, are served only through the authenticated, ownership-checked `GET /api/v1/uploads/{stored_name}` route, and have magic-byte content validation for binary types — there is no public `/static/uploads` mount.
- `/metrics` and `/runtime` require `require_role("admin")`.
- The rate limiter's Redis path fails open to an in-process fallback on a Redis outage, not closed to a full 429 outage.
- The audit log (`backend/observability/audit.py`) persists every entry to `logs/audit.jsonl` in addition to structured logging, and `user.deleted` entries correctly record the acting user.
- There is no stale duplicate rate limiter at `backend/common/rate_limit.py` — that module doesn't exist, and the test suite's cleanup fixture correctly resets `backend.resilience.rate_limit.shared_rate_limiter`.
- Login and registration both normalize email to `.strip().lower()` before lookup/compare — no case-sensitivity mismatch.
- `LocalUploadStorage.save()` returns a route path (`/api/v1/uploads/{stored_name}`), not a server filesystem path.
- `backend/core/security/rbac.py`'s `require_role()`/`AuthorizationPolicy` check `current_user.role`/`is_superuser`, not `username`.
- The initial migration's missing columns and mistyped timestamps are fixed by the follow-up migration `alembic/versions/20260707_add_missing_user_columns.py`; `alembic upgrade head` runs cleanly end-to-end (re-verified in this revision).
- Cloud storage backends for uploads (S3, Azure Blob) already exist in `backend/infrastructure/upload_storage.py` as `S3UploadStorage`/`AzureUploadStorage`, selected via `UPLOAD_BACKEND`, with config validation in `backend/core/config.py` requiring the matching credentials be present — this was previously listed only as roadmap/future work.

Several `tests/*` docstrings (e.g. in `test_authorization_and_roles.py`, `test_password_reset_and_email_verification.py`) still cite a file named `IMPROVEMENT.MD`/`IMPROVEMENT_SUGGESTIONS_MERGED.md` and describe some of the above as open bugs. That file doesn't exist in this repository — treat those docstrings as stale and due for a rewrite, not as a live issue list.

### 11.3 Test coverage note

Some tests assert that a package/module exists, imports successfully, or that a particular substring appears in a router's source, rather than exercising real behavior end-to-end. That's a useful smoke test, but a green suite alone isn't proof that realtime events, role checks, or endpoint auth are fully correct — prefer a behavioral test (real HTTP/Socket.IO client, real assertions on the response) going forward.

## 14. Billing, subscriptions, and entitlements

This module exists in full in the current source (`backend/domain/billing/`, `backend/application/billing/`, `backend/app/api/v1/billing/router.py`, `backend/core/security/entitlements.py`, `backend/integrations/stripe_adapter.py`/`paypal_adapter.py`) but was absent from every previous revision of this document — it was tracked only as a hypothetical roadmap item ("add a subscriptions module") despite already being implemented, tested (`tests/test_billing_hardening.py`, `tests/test_rbac_and_subscription_regression.py`), and mounted at `/api/v1/billing`. This section closes that gap.

### 14.1 Data model

- **`Plan`** (`plans` table) — `key` (unique), `name`, `price_cents`, `billing_interval` (default `month`), `is_active`.
- **`Feature`** (`features` table) — `key` (unique), `name`, `description`.
- **`PlanFeature`** (`plan_features` table) — a many-to-many join between plans and features; a plan "includes" a feature if a row exists.
- **`Subscription`** (`subscriptions` table) — links a `user_id` to a `plan_id`, with `status` (free-text, conventionally `active`/`trialing`/`canceled`) and `provider` (`manual`/`stripe`/`paypal`). A user can have multiple `Subscription` rows over time; routes that need "the current one" take the most recent by id (`order_by(Subscription.id.desc()).limit(1)`) rather than reading a dedicated "current subscription" pointer.
- **`PaymentEvent`** (`payment_events` table) — one persisted row per processed webhook event, unique on `(provider, provider_event_id)`. This is the durable half of webhook idempotency; `backend/resilience/idempotency.py`'s Redis-backed store is the fast-path half, checked first.

All four billing models live under `backend/domain/billing/`; `PaymentEvent` specifically is in `webhook_models.py` rather than `models.py`.

### 14.2 How entitlement checks work

`BillingService.user_has_feature(user, feature_key)` (`backend/domain/billing/service.py`) and its near-duplicate `BillingUseCases.user_has_feature(...)` (`backend/application/billing/use_cases.py`) both: return `True` immediately for `is_superuser`; otherwise collect the user's `active`/`trialing` `Subscription.plan_id`s; return `False` if there are none; otherwise check whether any of those plans has a `PlanFeature` row joined to a `Feature` with the given `key`.

Two call sites wrap this the same way:

- `GET /api/v1/billing/feature-check?feature=<key>` (route-level, returns a JSON error body).
- `require_feature("<key>")` (`backend/core/security/entitlements.py`, a reusable `Depends()` for gating _any other_ route — raises `HTTPException` instead of returning a response body directly).

Both short-circuit to "allowed" when `settings.subscriptions_enabled` is `False` (the default) or when the feature key isn't in `settings.disabled_features`; both return `402 Payment Required` with `{"error": "feature_not_in_plan", "required_feature": ..., "upgrade_url": "/billing/plans"}` when the plan lookup fails, and `503` with `{"error": "feature_disabled", ...}` when the key is force-disabled. No current route in this codebase actually uses `require_feature()` to gate itself yet — it exists as infrastructure for a future paid route, exercised directly by the regression tests rather than by, say, the products or uploads routers.

### 14.3 Checkout, webhooks, and idempotency

- `POST /billing/checkout` picks `StripeAdapter` or `PayPalAdapter` based on the `provider` field (defaulting to `stripe`), both implementing the same `PaymentGatewayPort` protocol (`backend/application/billing/use_cases.py`): `create_customer`, `create_checkout_session`, `verify_webhook_signature`.
- **Both adapters are mocks today.** `create_checkout_session` returns a deterministic fake `checkout_url`/`customer_id` (`f"cus_{user.id}"` / `f"pp_{user.id}"`) without an outbound HTTP call to Stripe or PayPal, and `verify_webhook_signature` treats any non-empty `signature_header` as verified. This is enough to exercise the full plan → checkout → webhook → subscription workflow in tests and local dev, but it is **not** connected to real payment processing — see §13 for the roadmap item to replace these with the real SDKs.
- `POST /billing/webhooks/stripe` is idempotent two ways: it first checks the Redis-backed idempotency store keyed on `f"{provider}:{event_id}"`, and — belt-and-suspenders — also checks for an existing `PaymentEvent` row with the same `(provider, provider_event_id)` before inserting a new one (the column pair has a real `UniqueConstraint`, so even a race between two idempotency-store misses can't double-insert). A duplicate delivery returns `{"processed": false, "duplicate": true, "event_id": ...}` without re-applying any side effect.
- On `event_type == "checkout.session.completed"`, the handler creates a new `active` `Subscription` for the `user_id`/`plan_id` in the payload. No other event types (e.g., `invoice.payment_failed`, `customer.subscription.deleted`) are handled yet — this is part of the "not yet a full lifecycle" gap noted in §13.

### 14.4 Admin surface and RBAC coupling

Every admin billing route under `/api/v1/billing/...` requires the `billing.manage` permission via `require_permission("billing.manage")` (`backend/core/security/rbac.py`), not a role name — a user needs that permission key in `current_user.permissions` (or `is_superuser`), independent of whatever `role` string they have. `RbacService.ensure_seed_data()` (`backend/domain/rbac/service.py`) seeds both `billing.manage` and `system.billing_toggle` permission keys on first use.

**`system.billing_toggle` is not unused — it gates a second, separate toggle route with the same effect as part of a different router.** `PATCH /api/v1/admin/system/subscriptions-enabled` (`backend/app/api/v1/admin/router.py`) flips the exact same `settings.subscriptions_enabled` flag as `PATCH /api/v1/billing/admin/settings`, but lives on the admin router, is gated by `system.billing_toggle` instead of `billing.manage`, and only touches `subscriptions_enabled` (not `payment_providers_enabled`). Both routes are real and both work; there's no conflict, but it means there are two independently-permissioned ways to flip the same setting, and neither route is currently exercised by the test suite. Worth consolidating onto one route/permission before relying on this in a real deployment — noted in §13.

### 14.5 Known gaps specific to this module

- `mrr_cents` in `GET /billing/admin/billing/metrics` sums `Subscription.plan_id` values, not `Plan.price_cents` — it is a response-shape placeholder, not a real revenue calculation. `churned_this_month` and `trial_conversion_rate` are hardcoded to `0`/`0.0`.
- There's no `backend/contracts/billing_contracts.py` — unlike `auth`/`health`/`products`/`users`/`uploads`, the billing router takes/returns raw `dict[str, object]` payloads rather than typed Pydantic contracts, so request validation is manual (`int(payload.get(...))`-style coercion) rather than FastAPI's usual 422-on-bad-input behavior.
- No trial-period expiry job, no proration on plan changes, and no invoice/payment-history record beyond the raw `PaymentEvent.payload` JSON blob.
